# Conflict Resolution — architecture (P0 baseline, 2026-06-12)

## The two universes

Two completely disjoint implementations of "conflict in a circle" coexist in
the codebase today. Neither talks to the other, and the user-facing surfaces
are split across them. This document exists so the next engineer doesn't
spend a day figuring out why one tab is empty and another is full of rows
that look identical.

### Universe A — AI Conflict Prediction (heuristic / ML)

| Layer | Where |
| --- | --- |
| Screens | `screens/ConflictAlertScreen.tsx` (3 tabs: Formation, Monitoring, History) |
| Hook | `hooks/useConflictPrediction.ts` — `useFormationReview`, `usePostFormationMonitor`, `useConflictHistory`, `useConflictActions`, `useConflictDashboard` |
| Engine | `services/ConflictPredictionEngine.ts` |
| Tables | `conflict_history`, `conflict_prediction_dashboard`, `circle_formation_flags`, `post_formation_monitor` |

The engine inspects friction signals (declines, late contributions, dispute
counts, etc.) and emits tiered findings: `clear`, `compatible`, `watch`,
`flag`, `separate`. Findings land in `conflict_history` (per-conflict rows)
and roll up into `conflict_prediction_dashboard`.

### Universe B — Elder + Dispute + Mediation (manual workflow)

| Layer | Where |
| --- | --- |
| Screens | `ElderDashboardScreen.tsx`, `MediationCaseScreen.tsx`, `MediationToolsScreen.tsx`, `BecomeElderScreen.tsx`, `ElderTrainingHubScreen.tsx` |
| Context | `context/ElderContext.tsx` (and `CirclesContext.tsx` for the member-side `reportMember` insert) |
| Tables | `disputes`, `dispute_cases`, `dispute_messages`, `mediation_cases`, `elder_profiles`, `elder_council_votes`, `elder_endorsements`, `elder_stats`, `elder_vote_records`, `elder_applications` |

A member taps "Report a member" in CircleDetail → `disputes.insert`. An
elder picks the row up via Elder Dashboard → opens MediationCase → posts
messages, votes, resolves → updates `mediation_cases.status`.

## What this means in practice

- A member-reported dispute is **invisible** to the AI dashboard. The History
  tab of `ConflictAlertScreen` reads `conflict_history`, not `disputes`.
- An AI-flagged formation issue is **invisible** to elders unless someone
  manually escalates it into a `disputes` row.
- The two tier vocabularies don't align: Universe A uses `clear/watch/flag/
  separate`, Universe B uses `open/in_review/escalated/resolved`.
- Notifications wired in P0 (this file) only fire on Universe B writes
  (`disputes.insert`). Universe A's writes are silent because they're
  generated by a server-side cron that runs without user interaction.

## P0 (shipped 2026-06-12)

1. Fixed `CirclesV2Screen.handleOpenConflict` — it was navigating to
   `Routes.ConflictAlert` without the required `circleId` param. The route
   signature is now `{ circleId?: string } | undefined`, and the screen
   renders an inline circle picker (or a no-circles empty state) when no
   id is provided.
2. Added architecture banners to both `useConflictPrediction.ts` and
   `ElderContext.tsx` cross-referencing this document so the split is
   discoverable from either entry point.
3. Wired `notifications.insert` into `CirclesContext.reportMember` so
   elders of the affected circle get a row in the notifications table when
   a new dispute is filed. Today this is best-effort and non-fatal; a real
   push delivery layer can replace the insert without changing the call
   sites.
4. Added a 5-minute module-level cache to `useConflictHistory` and
   `usePostFormationMonitor` (keyed by `memberId`/`circleId`) and wired
   `useFocusEffect` on the consumer screen. Same pattern as
   `useAdvanceDashboard` / `useScoreHub` / `useUpcomingEvents`.
